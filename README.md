# MOEX Large Order Alert Bot

Telegram-бот, который мониторит стакан заявок и графические паттерны по
топ-ликвидным акциям Московской биржи через **T-Invest API (Т-Банк)** —
без искусственной задержки, характерной для бесплатного MOEX ISS.

## Быстрый старт

```bash
python -m venv venv
source venv/bin/activate        # Windows: venv\Scripts\activate
pip install -r requirements.txt
```

Создайте файл `.env` рядом с `main.py` (см. `.env.example`) с тремя
значениями:

```
TELEGRAM_BOT_TOKEN=токен от @BotFather
TELEGRAM_CHAT_ID=ваш chat_id (узнать через @userinfobot)
T_INVEST_TOKEN=токен T-Invest API
```

**Как получить `T_INVEST_TOKEN`**: в приложении/личном кабинете
Т-Инвестиций -> Настройки -> Инвестиции -> Токен для API -> выпустить
токен с правами на чтение котировок (для этого бота торговые права не
нужны, только read-only). Подробнее:
https://developer.tbank.ru/invest/intro/intro

```bash
python main.py
```

## Источник данных: почему T-Invest, а не MOEX ISS

Изначально бот использовал бесплатный MOEX ISS API, но у него два
существенных ограничения для этой задачи:
1. Полный стакан заявок доступен только по платной подписке
2. Исторические данные (включая минутные свечи) отдаются с задержкой
   ~15 минут даже при наличии токена

T-Invest API (при наличии токена доступа к боевому контуру) отдаёт и
стакан, и свечи практически в реальном времени, что критично для
паттернов на 1-минутном таймфрейме. Старые модули (`moex_api.py`,
`candles_api.py`, `debug_candles.py`) оставлены в проекте только как
диагностический инструмент — основной бот их больше не использует
(см. `tinvest_client.py`).

### Почему без официальной Python-библиотеки

Официальный SDK `tinkoff-investments` на момент написания периодически
попадает в карантин на PyPI (вместе с одной из своих зависимостей),
из-за чего `pip install` может просто не сработать в произвольный
момент времени. Чтобы не зависеть от этого, `tinvest_client.py`
обращается к официальному **REST/JSON-шлюзу** T-Invest API напрямую
через `aiohttp` (тот же пакет, что уже используется в проекте) — без
gRPC, без protobuf, без сторонних SDK. Технически это тот же самый API,
просто без обёртки.

Задокументированный REST-протокол:
https://developer.tbank.ru/invest/intro/developer/protocols/restapi

### Российские TLS-сертификаты (НУЦ Минцифры)

T-Bank переходит на TLS-сертификаты российского удостоверяющего центра
(НУЦ Минцифры / Russian Trusted Root CA) — они не входят в стандартный
список доверенных сертификатов, который использует Python. Без
дополнительной настройки запросы к `invest-public-api.tbank.ru` будут
падать с `SSLCertVerificationError`, причём **одинаково что локально,
что в облаке** — это не сетевая проблема, а особенность самого
сертификата сервера.

Решение уже встроено в `tinvest_client.py` (функция `_build_ssl_context`),
но требует один раз вручную скачать два файла сертификатов с
официального источника (Минцифры/Госуслуги) и положить их в папку
`certs/` рядом с `main.py`:

1. https://gu-st.ru/content/Other/doc/russian_trusted_root_ca.cer → `certs/russian_trusted_root_ca.cer`
2. https://gu-st.ru/content/Other/doc/russian_trusted_sub_ca.cer → `certs/russian_trusted_sub_ca.cer`

Если файлов нет — бот не упадёт, но при обращении к T-Invest будет
выдавать `SSLCertVerificationError` и подсказывать в логе, что нужно
скачать эти файлы.

## Структура проекта

- `config.py` — тикеры, пороги детекции, токены.
- `tinvest_client.py` — **основной источник данных**: стакан и свечи через T-Invest API.
- `detector.py` — логика определения "крупной" заявки и фильтр от
  спуфинга (заявок, которые ставятся и почти сразу отменяются).
- `pattern_detector.py` — детекция графических паттернов на свечах:
  свечные паттерны, пробой уровней S/R, классические фигуры, объёмный
  импульс (подробности ниже).
- `risk_calculator.py` — расчёт цены входа, стоп-лосса и тейк-профита.
- `settings.py` — какие категории сигналов включены/выключены (управляется из Telegram, см. ниже), сохраняется в `settings.json`.
- `telegram_commands.py` — обработка команды `/settings` с inline-кнопками для включения/выключения категорий.
- `notifier.py` — форматирование, фильтрация по настройкам и отправка сообщений в Telegram.
- `main.py` — главный цикл: запускает три параллельных потока (стакан + свечи + приём команд Telegram) через T-Invest API.
- `moex_api.py`, `candles_api.py`, `debug_candles.py`, `debug_tinvest_time.py` — **legacy/диагностика**, основной бот их не использует.

## Управление сигналами прямо из Telegram

Отправьте боту команду `/settings` — появятся кнопки для каждой
категории сигналов (✅ включено / 🚫 выключено), нажатие переключает
состояние. Настройки сохраняются в `settings.json` рядом с `main.py` и
не сбрасываются при перезапуске бота.

Категории: крупные заявки, свечные паттерны, уровни S/R, классические
фигуры, объёмные всплески — то есть можно, например, оставить только
свечные паттерны и пробои уровней, отключив более шумные классические
фигуры (двойная вершина/дно, треугольники), не трогая код.


## Модуль графических паттернов (pattern_detector.py)

Работает на 1-минутных свечах (настраивается через
`PATTERN_CANDLE_INTERVAL` в `config.py`). Каждые `PATTERN_POLL_INTERVAL_SEC`
секунд запрашивает свежие свечи и проверяет последнюю закрытую свечу на:

1. **Свечные паттерны** (надёжная математика: соотношения тела/теней):
   доджи, молот, перевёрнутый молот/падающая звезда, бычье/медвежье
   поглощение.
2. **Пробой уровня поддержки/сопротивления**: уровни строятся по
   локальным пивотам истории (`PIVOT_WINDOW`), пробой засчитывается,
   если цена закрытия прошла уровень минимум на `SR_BREAKOUT_MIN_PCT`.
3. **Классические фигуры** (эвристика, ниже надёжности пунктов 1-2):
   двойная вершина/дно (два близких пивота + допуск
   `DOUBLE_TOP_BOTTOM_TOLERANCE_PCT`), сужающийся треугольник (грубая
   проверка на сходящиеся максимумы/минимумы). **Голова-плечи в текущей
   версии не реализована** — эта фигура требует более сложной логики
   (три пика с определённым соотношением высот), при необходимости
   можно добавить отдельно.
4. **Аномальный объём свечи**: объём свечи в `VOLUME_SPIKE_MULTIPLIER`
   раз больше среднего по истории.

### Честно про ограничения этого модуля

- Свечные паттерны и пробой уровней — это чистая математика по OHLC,
  детектируются надёжно как факт паттерна. Но сам факт появления
  паттерна **не гарантирует** движение цены в ожидаемую сторону —
  это статистическая тенденция, а не правило.
- Двойная вершина/дно и треугольники — не имеют единого строгого
  определения даже в классическом техническом анализе, поэтому любая
  автоматическая реализация — это чья-то интерпретация с параметрами
  допуска. Ожидайте больше ложных срабатываний по этим двум пунктам,
  чем по свечным паттернам.
- На 1-минутном таймфрейме шума физически больше, чем на часовом или
  дневном — некоторые "паттерны" будут просто рыночным шумом. Параметры
  в `config.py` (допуски, множители) — отправная точка, калибруйте под
  свои тикеры по факту работы.

## Расчёт цены входа, стоп-лосса и тейк-профита (risk_calculator.py)

Каждый сигнал (крупная заявка или паттерн) с ненулевым направлением
(bullish/bearish, BUY/SELL — не neutral) теперь сопровождается расчётом:

- **Вход** — цена на момент сигнала
- **Стоп-лосс** — для паттернов считается через ATR (Average True Range,
  среднее реальное движение цены за период `ATR_PERIOD` баров,
  умноженное на `ATR_STOP_MULTIPLIER`); для сигналов о крупных заявках
  (там нет истории свечей) — через фиксированный процент
  `LARGE_ORDER_STOP_LOSS_PCT`
- **Тейк-профит** — расстояние от входа до тейка = расстояние до стопа,
  умноженное на `RISK_REWARD_RATIO` (по умолчанию 2.0, то есть соотношение
  риск/прибыль 1:2)

**Это механический расчёт по фиксированным правилам, а не финансовая
рекомендация.** ATR и процентный стоп — это отправная точка технического
анализа, а не гарантия того, что сделка не пробьёт эти уровни на шумном
движении рынка. Расчёт не учитывает: текущие корпоративные события,
объём торгов в моменте, общий рыночный контекст, размер вашей позиции.
Обязательно проверяйте адекватность уровней на здравый смысл перед
использованием, особенно на низколиквидных бумагах или в моменты
повышенной волатильности (открытие/закрытие торгов, выход новостей).

Настройки в `config.py`: `ATR_PERIOD`, `ATR_STOP_MULTIPLIER`,
`RISK_REWARD_RATIO`, `LARGE_ORDER_STOP_LOSS_PCT`.

## Диагностика (debug_candles.py, legacy)

Этот скрипт обращается напрямую к MOEX ISS (не к T-Invest) — полезен,
если захотите сверить данные между источниками или устроить диагностику
независимо от T-Invest API:

```powershell
python debug_candles.py ТИКЕР
```

### Лимит запросов T-Invest API (100/мин) и как с ним жить

T-Invest документированно ограничивает unary-методы (`GetOrderBook`,
`GetCandles`) **100 запросами в минуту** на токен. Это делит бюджет
между количеством тикеров и частотой опроса — чем больше тикеров в
`config.TICKERS`, тем реже приходится их опрашивать, чтобы уложиться в
лимит.

Значения по умолчанию (`POLL_INTERVAL_SEC=15`, `PATTERN_POLL_INTERVAL_SEC=30`)
рассчитаны на 15 тикеров и укладываются примерно в 90 запросов/мин.
Если хотите более частый опрос — уменьшите список `TICKERS`.
Формула: `запросы/мин = (тикеров / POLL_INTERVAL_SEC + тикеров / PATTERN_POLL_INTERVAL_SEC) * 60`.

На случай кратковременных всплесков (или если решите опрашивать чаще
документированного лимита) `tinvest_client.py` содержит собственный
rate limiter и retry с backoff при HTTP 429 — но это подстраховка,
не замена правильно посчитанных интервалов.

## Как настраивать чувствительность

В `config.py`:

- `VOLUME_MULTIPLIER_THRESHOLD` — во сколько раз объём заявки должен
  превышать средний, чтобы считаться крупной. Начните с 8–10 и
  откалибруйте по факту (слишком много шума — увеличьте).
- `MIN_NOTIONAL_RUB` — отсекает крупные *в лотах*, но незначительные
  *в деньгах* заявки по низкоценным бумагам.
- `CONFIRM_STANDING_SEC` — фильтр от спуфинга. Чем больше — тем меньше
  ложных срабатываний, но тем позже придёт сигнал.
- `COOLDOWN_SEC` — защита от спама повторными уведомлениями по одному
  и тому же уровню цены.

## Важные ограничения текущей версии

1. **T-Invest ограничивает глубину стакана бесплатно тоже** — `depth=1`
   в `tinvest_client.py` даёт только лучшую цену bid/ask и объём по ней
   (аналогично тому, что было с MOEX ISS). Полную глубину рынка можно
   запросить (параметр `depth` до 50), но для детектора крупных заявок
   лучшая цена — как раз то, что нужно.
2. **REST-опрос — не полноценный стриминг.** По умолчанию бот опрашивает
   T-Invest раз в секунду (стакан) и раз в 15 секунд (свечи). Это заметно
   быстрее, чем было с MOEX ISS, но всё ещё не то же самое, что
   WebSocket-подписка на поток сделок. T-Invest API поддерживает
   потоковую подписку (`market_data_stream`) — при необходимости можно
   переписать `tinvest_client.py` на стриминг для минимальной задержки.
3. **Крупная заявка ≠ гарантированное движение цены.** Это может быть
   айсберг-ордер, заявка маркет-мейкера или заявка, которая будет
   отменена. Фильтр `CONFIRM_STANDING_SEC` снижает, но не убирает
   этот риск.
4. Бот **только уведомляет**, торговых операций не совершает.

## Планы на будущее: автоторговля

T-Invest API поддерживает не только чтение котировок, но и выставление
торговых поручений (`client.orders.post_order(...)`) с тем же токеном
(если у него есть торговые права, а не только read-only). Это означает,
что для перехода от уведомлений к (полу)автоматической торговле **не
обязательно** подключать QUIK — можно остаться в той же экосистеме
T-Invest API, что упрощает архитектуру.

Если всё же понадобится QUIK (например, ВТБ) — ниже оставлен план,
актуальный независимо от источника рыночных данных:

Когда будете готовы перейти от уведомлений к (полу)автоматической
покупке, естественный путь:

1. **QuikSharp** или **QUIK LUA (qlua)** — скрипт на LUA внутри QUIK,
   который публикует стакан/заявки через локальный сокет/файл, либо
   Python-обёртки типа `QuikPy`, которые общаются с QUIK через DDE/
   QuikSharp-мост.
2. Для получения сделок в реальном времени из QUIK обычно поднимают
   локальный сервис (LUA-скрипт → сокет → ваш Python-процесс), который
   публикует события в ту же шину, что сейчас использует `detector.py`
   — тогда `detector.py` и `notifier.py` можно переиспользовать почти
   без изменений, поменяется только источник данных (`moex_api.py`
   будет заменён/дополнен модулем `quik_api.py`).
3. Для выставления заявок на покупку через QUIK — тоже через
   QLUA-скрипт (`sendTransaction`) или через QuikSharp API.
4. Т.к. это уже реальные торговые операции с реальными деньгами,
   потребуется отдельный слой риск-менеджмента (лимиты на сделку,
   стоп-лоссы, дневной лимит потерь) — рекомендую не пропускать этот
   шаг при переходе от уведомлений к исполнению ордеров.

Дайте знать, когда будете готовы к этому этапу — структура проекта уже
сделана так, чтобы добавить исполнение заявок было относительно легко.
